# Plan 工作流

前提：docs/plans/ 下必须有已批准的设计文档。

## 步骤
1. 读取设计文档，完全理解需求
2. 用 subagent 扫描相关代码（scripts/、src/、skills/），做 gap analysis：
   - 哪些功能已实现、哪些部分实现、哪些完全缺失
   - 现有代码中可复用的模块/函数
   - 潜在冲突点（同一文件被多个任务修改）
3. 将工作拆成小任务，每个任务包含：
   - 要改的具体文件路径
   - 实现逻辑描述（不需要完整代码，但要足够清晰）
   - 验证步骤：跑什么命令、期望什么结果
   - **失败回退策略**：如果这个任务卡住了，降级方案是什么（例：换一种实现方式、先跳过用 hardcode、拆成更小子任务）
4. 按依赖关系和优先级排序
5. 保存为 docs/plans/YYYY-MM-DD-<feature>-impl-plan.md
6. git commit

## 任务粒度标准
- 每个任务能在一次 Claude Code 交互中完成
- Pipeline 类任务：一次改一个 stage，不要跨 stage
- 前端任务：一次改一个页面或组件
- 如果说不清 "验证步骤"，说明任务还需要继续拆

## Subagent 规则
- **代码扫描 / gap analysis** → 派 subagent 做。给 subagent 明确指令：扫描哪些目录、查找什么模式、返回什么格式的摘要。
- **写实现计划** → 主 agent 自己做（需要全局视角和上下文连贯性）。
- **检查已有测试覆盖** → 派 subagent 扫描 `__tests__/` 目录，返回已有测试列表和覆盖的功能点。

## 不做的事
- 不写任何实现代码
- 不做 "假设没实现" 的判断——先搜索代码确认
